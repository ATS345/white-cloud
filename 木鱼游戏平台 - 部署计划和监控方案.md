# 木鱼游戏平台 - 部署计划和监控方案

## 1. 概述

本文档详细描述了木鱼游戏平台的部署计划和监控方案，包括部署环境、部署工具、部署流程、监控体系、日志管理、备份与恢复、安全措施等内容。本计划旨在确保平台的可靠性、可扩展性和安全性，实现高效的部署和监控管理。

## 2. 部署环境

### 2.1 环境分类

| 环境 | 用途 | 配置 |
|------|------|------|
| 开发环境 | 开发人员日常开发和自测 | 本地开发机器 |

| 生产环境 | 正式对外服务 | 生产服务器集群（6台服务器） |

### 5.2 镜像管理

#### 5.2.1 镜像仓库

- **私有镜像仓库**：Harbor
- **镜像标签策略**：
  - 开发环境：`{service-name}:{commit-sha}`
  - 测试环境：`{service-name}:{branch-name}`
  - 预发布环境：`{service-name}:{version}-rc.{build-number}`
  - 生产环境：`{service-name}:{version}`

#### 5.2.2 镜像构建

- **构建工具**：Dockerfile
- **构建策略**：多层构建，减小镜像体积
- **基础镜像**：Alpine Linux
- **镜像扫描**：使用Trivy扫描镜像漏洞

### 5.3 部署策略

#### 5.3.1 滚动更新

- **更新策略**：滚动更新，逐步替换旧的Pod
- **最大不可用Pod数**：25%
- **最大额外Pod数**：25%
- **更新间隔**：30秒

#### 5.3.2 蓝绿部署

- **适用场景**：重大版本更新
- **部署步骤**：
  1. 部署新的服务版本（蓝环境）
  2. 测试新环境
  3. 切换流量到新环境
  4. 监控新环境
  5. 确认无误后删除旧环境（绿环境）

#### 5.3.3 金丝雀部署

- **适用场景**：新功能测试
- **部署步骤**：
  1. 部署少量新的Pod（10%流量）
  2. 监控新Pod的性能和错误率
  3. 逐步增加新Pod的数量（25% → 50% → 75% → 100%）
  4. 确认无误后删除旧Pod

### 6.4 告警规则

#### 6.4.1 告警级别

| 级别 | 定义 | 通知方式 |
|------|------|----------|
| 紧急（P0） | 系统崩溃、数据丢失、核心功能无法使用 | 电话、短信、邮件、企业微信 |
| 高（P1） | 重要功能无法使用、影响大量用户 | 短信、邮件、企业微信 |
| 中（P2） | 次要功能无法使用、影响部分用户 | 邮件、企业微信 |
| 低（P3） | 轻微问题、不影响用户体验 | 企业微信 |

#### 6.4.2 告警规则示例

```yaml
# 服务实例数不足
- alert: ServiceInstanceCountLow
  expr: kube_deployment_status_replicas_available{namespace="default"} < kube_deployment_spec_replicas{namespace="default"}
  for: 5m
  labels:
    severity: P1
  annotations:
    summary: "服务实例数不足"
    description: "服务 {{ $labels.deployment }} 的可用实例数 ({{ $value }}) 低于预期实例数"

# 请求成功率低
- alert: ServiceRequestSuccessRateLow
  expr: sum(rate(http_requests_total{status=~"5.."}[5m])) by (service) / sum(rate(http_requests_total[5m])) by (service) > 0.01
  for: 5m
  labels:
    severity: P1
  annotations:
    summary: "服务请求成功率低"
    description: "服务 {{ $labels.service }} 的错误率 ({{ $value | humanizePercentage }}) 超过1%"

# 平均响应时间长
- alert: ServiceResponseTimeHigh
  expr: histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (service, le)) > 0.5
  for: 5m
  labels:
    severity: P2
  annotations:
    summary: "服务响应时间长"
    description: "服务 {{ $labels.service }} 的P95响应时间 ({{ $value }}秒) 超过500ms"

# CPU使用率高
- alert: NodeCPUUsageHigh
  expr: (100 - (avg by (instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)) > 80
  for: 5m
  labels:
    severity: P2
  annotations:
    summary: "服务器CPU使用率高"
    description: "服务器 {{ $labels.instance }} 的CPU使用率 ({{ $value }}%) 超过80%"

# 内存使用率高
- alert: NodeMemoryUsageHigh
  expr: (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes * 100 > 85
  for: 5m
  labels:
    severity: P2
  annotations:
    summary: "服务器内存使用率高"
    description: "服务器 {{ $labels.instance }} 的内存使用率 ({{ $value }}%) 超过85%"
```

## 7. 日志管理

### 7.1 日志分类

| 日志类型 | 用途 | 存储周期 |
|---------|------|----------|
| 应用日志 | 应用程序运行日志 | 30天 |
| 访问日志 | API请求日志 | 7天 |
| 错误日志 | 错误和异常日志 | 90天 |
| 审计日志 | 操作审计日志 | 180天 |
| 数据库日志 | 数据库运行日志 | 15天 |
| 系统日志 | 服务器系统日志 | 7天 |

### 7.2 日志收集

- **收集工具**：Filebeat
- **收集方式**：
  - 应用日志：通过stdout输出到Docker日志
  - 系统日志：直接从文件收集
  - 数据库日志：直接从文件收集
- **日志格式**：JSON格式，包含时间戳、日志级别、服务名称、请求ID、消息内容等字段

### 7.3 日志存储

- **存储工具**：Elasticsearch
- **存储策略**：
  - 热数据：最近7天，存储在SSD磁盘
  - 温数据：7-30天，存储在HDD磁盘
  - 冷数据：30天以上，存储在对象存储
- **索引策略**：按天创建索引，自动滚动
- **压缩策略**：使用LZ4压缩

### 7.4 日志分析

- **分析工具**：Kibana
- **分析功能**：
  - 日志查询和过滤
  - 日志可视化
  - 日志告警
  - 日志关联分析
  - 日志导出

### 7.5 日志查询示例

```
# 查询特定服务的错误日志
GET /app-logs-*/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "service": "user-service" } },
        { "match": { "level": "ERROR" } },
        { "range": { "@timestamp": { "gte": "now-24h" } } }
      ]
    }
  },
  "sort": [ { "@timestamp": { "order": "desc" } } ]
}

# 查询请求响应时间超过1秒的日志
GET /access-logs-*/_search
{
  "query": {
    "bool": {
      "must": [
        { "range": { "response_time": { "gt": 1 } } },
        { "range": { "@timestamp": { "gte": "now-24h" } } }
      ]
    }
  },
  "sort": [ { "response_time": { "order": "desc" } } ]
}
```

### 8.2 数据恢复

#### 8.2.1 恢复流程

1. **恢复准备**：
   - 确定恢复点
   - 准备恢复环境
   - 停止相关服务

2. **数据恢复**：
   - 恢复数据库备份
   - 恢复配置备份
   - 恢复存储数据

3. **恢复验证**：
   - 启动相关服务
   - 验证数据完整性
   - 验证服务可用性
   - 验证业务功能

4. **恢复完成**：
   - 切换流量到恢复环境
   - 监控系统运行状态
   - 记录恢复过程

## 9. 安全措施

### 9.1 网络安全

- **防火墙**：配置严格的防火墙规则，只允许必要的端口和IP访问
- **WAF**：部署Web应用防火墙，防止SQL注入、XSS、CSRF等攻击
- **DDoS防护**：部署DDoS防护服务，防止DDoS攻击
- **网络隔离**：使用VLAN和网络策略隔离不同网络区域
- **加密传输**：所有内部和外部通信使用HTTPS/TLS 1.3

### 9.2 访问控制

- **身份认证**：使用OAuth 2.0和JWT进行身份认证
- **基于角色的访问控制（RBAC）**：实现细粒度的权限控制
- **最小权限原则**：每个用户和服务只授予必要的权限
- **多因素认证**：对管理员和敏感操作启用多因素认证
- **API密钥管理**：定期轮换API密钥，使用哈希存储

### 9.3 容器安全

- **镜像安全**：
  - 使用官方或经过验证的基础镜像
  - 定期扫描镜像漏洞
  - 签名和验证镜像
  - 最小化镜像体积

- **容器运行时安全**：
  - 使用只读文件系统
  - 限制容器权限
  - 隔离容器网络
  - 监控容器行为

### 9.4 数据安全

- **数据加密**：
  - 传输加密：HTTPS/TLS 1.3
  - 存储加密：AES-256
  - 敏感数据加密：使用KMS加密敏感数据

- **数据脱敏**：
  - 日志中的敏感数据脱敏
  - 测试数据脱敏
  - 数据导出脱敏

- **数据销毁**：
  - 定期销毁过期数据
  - 安全删除数据，防止恢复

### 9.5 安全审计

- **操作审计**：记录所有管理操作和敏感操作
- **审计日志**：存储在独立的安全日志系统，防止篡改
- **审计分析**：定期分析审计日志，发现异常操作
- **合规报告**：生成合规报告，满足监管要求

## 10. 自动化运维

### 10.1 自动化部署

- **CI/CD自动化**：通过GitLab CI和Argo CD实现自动化部署
- **基础设施自动化**：使用Terraform和Ansible实现基础设施自动化
- **配置管理自动化**：使用Ansible和Kubernetes ConfigMap实现配置管理自动化

### 10.2 自动化监控

- **自动发现**：自动发现新的服务和实例
- **自动配置**：自动配置监控指标和告警规则
- **自动告警**：根据监控数据自动发送告警
- **自动恢复**：对于某些常见故障，实现自动恢复

### 10.3 自动化扩缩容

- **水平扩缩容**：基于CPU使用率、内存使用率、QPS等指标自动扩缩容
- **扩缩容策略**：
  - 扩容阈值：CPU使用率 > 70% 或 内存使用率 > 80% 或 QPS > 1000
  - 缩容阈值：CPU使用率 < 30% 且 内存使用率 < 40% 且 QPS < 200
  - 扩缩容步长：每次增加或减少1个Pod
  - 扩缩容冷却时间：5分钟

### 10.4 自动化备份

- **自动备份**：按照备份策略自动执行备份
- **备份验证**：自动验证备份的完整性和可恢复性
- **备份监控**：监控备份任务的执行状态
- **备份告警**：备份失败时发送告警

## 11. 应急响应

### 11.1 应急响应流程

1. **事件检测**：通过监控系统或用户报告发现故障
2. **事件分级**：根据故障影响范围和严重程度分级
3. **应急响应**：
   - 成立应急响应小组
   - 隔离故障系统
   - 分析故障原因
   - 实施修复方案
   - 验证修复效果
4. **事件恢复**：恢复系统正常运行
5. **事件总结**：
   - 编写故障报告
   - 分析根本原因
   - 制定改进措施
   - 组织经验分享

### 11.2 常见故障处理

| 故障类型 | 处理步骤 |
|---------|---------|
| 服务不可用 | 1. 检查服务状态<br>2. 检查Pod状态<br>3. 检查日志<br>4. 重启服务或扩容<br>5. 验证服务可用性 |
| 数据库连接失败 | 1. 检查数据库状态<br>2. 检查网络连接<br>3. 检查数据库日志<br>4. 恢复数据库<br>5. 验证数据库连接 |
| 高CPU使用率 | 1. 检查进程占用<br>2. 检查应用日志<br>3. 优化代码或扩容<br>4. 验证CPU使用率 |
| 内存泄漏 | 1. 检查内存使用趋势<br>2. 分析内存快照<br>3. 修复内存泄漏<br>4. 重启服务<br>5. 验证内存使用率 |
| 网络故障 | 1. 检查网络连接<br>2. 检查防火墙规则<br>3. 检查DNS配置<br>4. 修复网络故障<br>5. 验证网络连接 |
